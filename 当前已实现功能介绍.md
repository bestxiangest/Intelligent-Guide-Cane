# 星辰引路者当前已实现功能介绍

本文档记录当前仓库已经打通的系统能力、模块边界和联调状态，便于演示、复现和后续迭代。

## 1. 系统组成

当前项目由四个端组成：

- ESP32-S3 主控固件：导盲杖本体，负责唤醒、录音、ASR/TTS、传感器、安全提醒和服务端通信。
- Flask 服务端：低延迟语音中枢，负责 fast intent、JSON 意图识别、函数分发、导航、天气、位置和状态管理。
- 微信小程序：地图、位置展示和 ESP32-CAM 视频预览入口。
- ESP32-CAM：独立摄像头辅助端，提供健康检查、抓拍和视频流。

## 2. 低延迟语音交互

已打通流程：

1. 用户按下语音按钮或说出唤醒词。
2. ESP32-S3 播放 LittleFS 中的 `record_start_001.wav`。
3. ESP32-S3 以 16kHz 采集用户语音，保留静音检测，最长录音 10 秒。
4. ESP32-S3 调用百度 ASR 得到文本。
5. ESP32-S3 发送文本到服务端 `POST /ai`，并携带 `X-Device-ID`。
6. 服务端优先执行 fast intent。
7. fast intent 无法判断时，调用 DashScope / 千问输出严格 JSON 意图。
8. 服务端校验 JSON、过滤低置信度和未知 intent。
9. dispatcher 根据白名单 intent 执行业务函数。
10. ESP32-S3 根据统一响应中的 `speak` 字段播放本地音频或回退百度 TTS。

简单问候、退出导航、常见天气查询、导航开始和导航状态查询不再总是调用 LLM。

## 3. 服务端能力

服务端已经拆分为 Flask Blueprint + service/client/model/util 结构：

```text
web/
├─ app.py
├─ config.py
├─ routes/
├─ services/
├─ clients/
├─ models/
├─ utils/
├─ tests/
└─ API.md
```

核心服务：

- `intent_service.py`：fast intent、LLM JSON 意图识别、JSON 修复和校验。
- `dispatcher.py`：白名单函数分发。
- `navigation_service.py`：导航会话、路线规划、更新、状态、退出和超时处理。
- `weather_service.py`：天气查询和语音短句生成。
- `location_service.py`：GPS 缓存、逆地理编码、位置过期判断。
- `response_service.py`：统一响应结构和本地音频协议。

## 4. JSON 意图协议

LLM 只允许输出如下 JSON：

```json
{
  "intent": "navigation.start",
  "confidence": 0.95,
  "arguments": {
    "destination": "图书馆"
  },
  "reply": "",
  "safety_level": "normal"
}
```

支持的 intent：

- `navigation.start`
- `navigation.exit`
- `navigation.update`
- `navigation.status`
- `weather.query`
- `chat.normal`
- `location.status`
- `device.status`
- `system.help`
- `unknown`

服务端不会直接信任模型返回的任意函数名，所有真实操作都必须经过 dispatcher 白名单。

## 5. fast intent 快速通道

已实现规则：

- 问候：`你好`、`您好`、`在吗`、`小助手`、`喂`
- 退出导航：`停止导航`、`取消导航`、`退出`、`不去了` 等
- 开始导航：`导航到...`、`带我去...`、`我要去...`、`帮我导航到...`
- 天气查询：包含 `天气`、`下雨`、`温度`、`冷不冷`、`热不热`
- 导航状态/更新：`还有多远`、`到哪了`、`快到了吗`、`下一步`、`怎么走`
- 帮助：`帮助`、`怎么用`、`你能做什么`

命中 fast intent 时，服务端直接返回结构化响应，不调用大模型。

## 6. 导航能力

已实现：

- 使用最新 GPS 坐标作为起点。
- GPS 不可用时使用默认坐标。
- 高德地理编码目的地。
- 高德步行导航规划路线。
- 按 `device_id` 维护导航会话。
- 支持导航开始、更新、状态查询、退出导航。
- 导航会话超时清理。
- 返回适合语音播报的下一步指令、剩余距离和预计时间。

ESP32-S3 仍保留导航状态同步，服务端返回 `navigation.active=true` 时本地进入导航状态，退出或完成时恢复待机。

## 7. 天气能力

已实现：

- 查询今天、明天、后天三类日期。
- `city=current` 时优先使用最近 GPS 逆地理编码得到的城市。
- 没有 GPS 城市时使用 `DEFAULT_WEATHER_CITY`。
- 返回适合语音播报的短句，例如“南昌市今天中雨，13 到 23 度，东北风 1-3 级。”

## 8. ESP32-S3 主控能力

已实现：

- 唤醒词触发和物理按钮触发。
- 按下按钮或唤醒后立即播放本地 `record_start_001.wav`。
- 播放提示音时暂停唤醒采集，避免回声和 I2S 资源冲突。
- 正式录音时暂停后台唤醒采集，避免抢占麦克风 I2S。
- 16kHz 录音，最长 10 秒，保留静音检测。
- 百度 ASR。
- 百度 TTS，网络返回音频按 16kHz 播放。
- LittleFS 本地音频缓存，所有缓存音频为 16kHz mono 16-bit PCM WAV。
- `AppState` 状态机：`BOOTING`、`WIFI_CONNECTING`、`IDLE`、`LISTENING`、`RECORDING`、`ASR_PROCESSING`、`SERVER_PROCESSING`、`SPEAKING`、`NAVIGATING`、`ERROR_STATE`。
- `DEVICE_ID` 设备标识。
- 请求服务端时统一发送 `X-Device-ID`。
- 兼容旧 `response` 字段，同时支持新 `speak` 字段。

## 9. 本地安全能力

以下能力继续在 ESP32-S3 本地运行，不依赖网络或大模型：

- 超声波避障检测。
- 震动模块提醒。
- 蜂鸣器提醒。
- 光敏照明控制。
- 未来跌倒检测、紧急呼叫等也应保持本地优先。

## 10. 本地音频缓存

服务端可返回：

```json
{
  "speak": {
    "mode": "local_audio",
    "audio_id": "hello_001",
    "text": "我在呢，请说导航、天气或者帮助。"
  }
}
```

ESP32-S3 播放策略：

1. `local_audio`：优先播放 LittleFS 中的 `audio_id.wav`。
2. 本地音频不存在时，回退到 `speak.text` 或 `response` 的百度 TTS。
3. `tts_text`：直接百度 TTS 播报文本。
4. `audio_url`：第一阶段预留，当前回退文本。
5. `none`：不播报。

当前缓存文件包括：

- `record_start_001.wav`
- `hello_001.wav`
- `wait_001.wav`
- `not_understood_001.wav`
- `nav_exit_001.wav`
- `no_navigation_001.wav`
- `network_error_001.wav`
- `gps_invalid_001.wav`
- `arrived_001.wav`
- `asr_empty_001.wav`
- `nav_starting_001.wav`
- `nav_planning_failed_001.wav`
- `server_busy_001.wav`
- `weather_error_001.wav`

## 14. 当前限制和后续方向

当前限制：

- 第一阶段仍由 ESP32-S3 调用百度 ASR/TTS。
- `/v1/audio` 只是预留入口，尚未接入 WebSocket 流式音频。
- 导航更新仍以服务端重新规划/更新为主，后续可做路线缓存和节流。
- ESP32-CAM 与小程序仍是相对独立链路，尚未与语音中枢深度融合。

后续方向：

- 服务端 ASR/TTS 接管。
- WebSocket 音频流上传。
- 流式 TTS 下发。
- 导航路线缓存。
- 跌倒检测和紧急呼叫。
- 更完善的设备状态上报。
